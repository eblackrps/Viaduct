Why now

Built for the teams that have to understand the estate before they can move it.

Viaduct is strongest when the hard part is not just copying disks. It is deciding what exists, what depends on what, what should move first, and what evidence an operator needs before a pilot wave is approved.

Use case

VMware-exit assessment

Inventory the estate, surface dependency clusters, and shape a first-wave candidate set before a team commits to a migration sequence.

Use case

Mixed-estate normalization

Bring VMware, Proxmox, Hyper-V, KVM, Nutanix, and Veeam context into one operator view instead of bouncing between disconnected tools.

Use case

Supervised pilot planning

Persist assumptions, approvals, saved plans, notes, and report exports so the assessment can become a reviewable pilot record.

Scope discipline

What the site is intentionally not claiming

  • Viaduct is not presented as general-availability automation.
  • The strongest current path is assessment, readiness, and supervised pilot execution.
  • Live migration behavior should still be validated in the operator's own lab or pilot environment.

Workflow

The first-run path is explicit, reviewable, and built around one persisted workspace.

The current dashboard experience starts with runtime authentication, then moves through a single workspace record that keeps discovery, graphing, readiness, plan state, and report export connected.

01

Bootstrap access

Use a service-account key for the normal operator path or a tenant key for bootstrap and break-glass administration.

02

Create workspace

Start with a named assessment record that holds source details, target assumptions, notes, and plan controls.

03

Discover and inspect

Persist source snapshots and dependency graph outputs so the operator can review the same state across refreshes and tools.

04

Simulate readiness

Run readiness analysis and recommendations against the same workspace instead of treating simulation as a detached report.

05

Save a dry-run plan

Persist plan state through the existing migration model so saved pilot decisions survive page changes and review loops.

06

Export evidence

Keep approvals, notes, and report exports attached to the same workspace so the pilot handoff is traceable.

Architecture

One stack, four engines, and one persisted operator model.

Viaduct keeps the current architecture intact: Go backend, REST API, CLI, React dashboard in web/, and a separate static public site in site/. The value is in the shared model and operator flow, not in adding more surface area.

Connectors

Read-only discovery first

Connectors normalize source and infrastructure metadata into the shared model before planning logic or execution decisions happen.

Core engines

Discovery, graph, migrate, lifecycle

The repo keeps its four-engine structure so discovery, dependency mapping, migration planning, and lifecycle analysis remain reviewable and separable.

Persistence

Workspace and job state survives refreshes

PostgreSQL is the recommended backend for any non-demo environment. The in-memory store remains the fast local path for the lab flow.

Operator surfaces

CLI, API, and dashboard share the same truth

The dashboard is not a separate demo shell. It reflects the same tenant-scoped state used by the API and CLI.

Platform coverage

Connector coverage matters because mixed estates are the norm. The stronger product story, though, is the assessment-to-pilot control plane wrapped around that coverage.

Evaluate

The cleanest path starts with the local lab, then moves outward deliberately.

Fresh clone to first successful workspace and report flow should be reproducible from source. The repo keeps the lab path, docs, API contract, and release verification aligned around that expectation.

Recommended order

1. Start local

Use examples/lab/, seed the lab tenant and service account, and run the workspace-first flow without requiring a live estate.

Recommended order

2. Review the contract

Check the REST surface in the published OpenAPI file and keep the current workflow grounded in the documented routes.

Recommended order

3. Move into a pilot carefully

Switch to PostgreSQL, keep service-account access explicit, and validate connector-specific behavior in your own supervised environment.
