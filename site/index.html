Why Viaduct

Built for the operators who need migration clarity before they commit to execution.

In mixed estates, the hard part is rarely just moving bits. Operators need current inventory, dependency context, planning discipline, and evidence they can hand to the next review step.

Mixed-estate visibility

Normalize inventory across platforms

Bring VMware, Proxmox, Hyper-V, KVM, Nutanix, and Veeam context into one operator surface instead of working from disconnected inventories.

Dependency context

See what should move together

Use network, storage, and backup relationships to understand clustering, sequencing, and risk before the first pilot workload is chosen.

Handoff-ready evidence

Keep notes, plans, and exports aligned

Persist the workspace record so discovery, simulation, saved plans, approvals, and exported reports stay tied to the same operator story.

Core capabilities

Professional operator surfaces built on the existing CLI, API, and shared backend model.

Viaduct keeps the current stack intact: Go backend, REST API, CLI, React dashboard in web/, and a static public site in site/. The product value comes from the shared model and disciplined workflow.

Discovery

Inventory normalization

Collect mixed-platform inventory and infrastructure metadata into one operator-visible schema before planning begins.

Analysis

Dependency mapping

Build workload, network, storage, and backup graph context so planning stays grounded in the actual estate.

Planning

Migration planning and supervised execution

Use declarative specs, readiness checks, saved plans, checkpoints, and operator-controlled execution instead of frontend-only assumptions.

Operations

Reports, auditability, and tenant context

Keep request correlation, service-account permissions, exported reports, and tenant-scoped operations visible across the dashboard and API.

Open source and operator controlled

Designed for technical evaluation, supervised pilots, and reviewable change control.

  • The repo ships the CLI, API, dashboard, docs, examples, release packaging, and published OpenAPI contract together.
  • PostgreSQL is the recommended persistent backend outside the demo path, while the lab stays fast with the in-memory store.
  • make release-gate remains the canonical verification path for backend, frontend, packaging, and smoke coverage.

Operator workflow

A guided workflow from bootstrap through report export.

The current operator experience starts with runtime authentication, then moves through one workspace record that keeps discovery, dependency analysis, readiness, plan state, and report export connected.

01

Bootstrap access

Use a service-account key for normal operator work or a tenant key for bootstrap and administrative recovery.

02

Create a workspace

Start with a named assessment record that holds the source connection, target assumptions, plan defaults, and operator notes.

03

Discover and inspect

Persist snapshots and dependency outputs so the same evidence survives refreshes, retries, and support handoffs.

04

Simulate readiness

Review readiness, policy issues, and remediation signals against the same workspace instead of detached reports.

05

Save the plan

Persist dry-run migration state through the backend migration model so operator review stays reproducible.

06

Export evidence

Attach reports, notes, and approval history to the same workspace so pilot review stays traceable.

Platform coverage

Coverage for the mixed environments operators actually inherit.

Viaduct focuses on mixed hypervisor estates where discovery, planning, and operator visibility matter as much as the eventual move itself.

VMware

vSphere discovery and migration planning inputs.

Proxmox

REST-backed discovery and a common target in current pilot examples.

Hyper-V

WinRM and PowerShell-backed collection.

KVM

Fixture-backed local lab plus optional live libvirt support.

Nutanix

Prism Central v3 discovery and planning visibility.

Veeam

Backup discovery, restore-point correlation, and portability context.

Community plugins

gRPC-based extension path with compatibility and health checks.

Architecture

One shared model across connectors, jobs, plans, and reports.

The architecture stays intentionally straightforward: connectors normalize source data, the core engines build planning context, and the operator surfaces share the same persisted backend truth.

Connectors

Read-only collection first

Platform-specific connectors normalize source metadata into the shared model before planning or execution logic is applied.

Core engines

Discovery, graph, migrate, lifecycle

The four-engine structure keeps discovery, dependency analysis, migration planning, and lifecycle analysis reviewable and separable.

Persistence

Workspace and job state survives refreshes

Persisted workspaces, job history, notes, and report exports stay available across retries, refreshes, and operator handoff.

Operator surfaces

CLI, API, and dashboard share the same truth

The dashboard is not a separate demo shell. It reflects the same tenant-scoped state used by the API and CLI.

Get started

The cleanest path starts local, then moves outward deliberately.

Fresh clone to first successful workspace and report flow should be reproducible from source. The local lab, operator docs, API contract, and release packaging stay aligned around that expectation.

Recommended order

1. Start with the lab

Use examples/lab/, seed the lab tenant and service account, and run the workspace-first flow without a live estate.

Recommended order

2. Review the operator contract

Use the published OpenAPI file and repo docs to keep the workflow grounded in the documented routes and packaged behavior.

Recommended order

3. Move into a supervised pilot

Switch to PostgreSQL, keep service-account access explicit, and validate connector-specific behavior in your own controlled environment.
