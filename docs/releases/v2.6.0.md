# Viaduct v2.6.0 Release Notes

## Upgrade Notes

- dashboard request dedupe is now explicit. Callers that relied on implicit GET dedupe should pass `{ dedupe: true }` when they want URL-based request coalescing.
- PostgreSQL duplicate-credential startup failures now exit as configuration errors and list the affected tenant IDs. See `docs/operations/credential-migration.md` before restarting.
- loopback-only runtime mode now emits explicit `AUDIT` rejection logs for rejected mutating requests, including the reason, peer, and forwarded-header presence.
- release artifact verification now pins `cosign verify-blob` to the exact tagged workflow identity for the release instead of a broad repository regex.

## Highlights

- fixed the remaining credential-compare timing edge so malformed stored hashes stay on a constant-cost path before the final constant-time compare
- made auth-session revocation atomic across PostgreSQL and the in-memory session manager, and immediately revoke sessions that no longer match the rotated tenant or service account credential
- hardened forwarded-address parsing for zoned and non-canonical inputs while preserving strict loopback enforcement even when trusted proxies are configured broadly
- added a distinct workspace enqueue timeout, stronger worker-shutdown handoff, and stress coverage for cancellation during heavy enqueue pressure
- taught the dashboard API client to resolve beneath a document `<base>` path, preserve external abort reasons, and use UUID-based in-flight tracking with opt-in request dedupe
- tightened focus-trap cleanup so dialogs recover focus safely even when the original trigger disappears mid-lifecycle

## Breaking Changes

- implicit GET request dedupe is no longer the default in the dashboard client
- forwarded addresses with IPv6 zones or non-canonical `X-Forwarded-For` entries are now rejected instead of being normalized loosely
- duplicate credential-hash startup failures are now treated as explicit configuration errors with exit code `78`
