# Viaduct v3.3.0 Release Notes

Viaduct `v3.3.0` is a published release focused on making the local Docker path usable from a fresh checkout: start the stack, open the dashboard, create an assessment, and run discovery without pasting a tenant or service-account key.

## Install

Use the signed GHCR image as the main packaged install path:

```bash
docker pull ghcr.io/eblackrps/viaduct:3.3.0
cosign verify ghcr.io/eblackrps/viaduct:3.3.0 \
  --certificate-identity \
  'https://github.com/eblackrps/Viaduct/.github/workflows/image.yml@refs/tags/v3.3.0' \
  --certificate-oidc-issuer \
  'https://token.actions.githubusercontent.com'
```

The Docker Hub mirror is available as `docker.io/emb079/viaduct:3.3.0` when repository secrets published it.

## What Changed

- The repo root now includes a local-only `compose.yaml` that starts PostgreSQL plus Viaduct on `127.0.0.1:8080` with no tenant key, service-account key, admin key, or copied secret config.
- The container image now includes the bundled KVM lab fixtures, so discovery works immediately in the dashboard local assessment workflow.
- The dashboard starts a trusted local operator session automatically when the server is in local-runtime mode, while service-account and tenant-key auth remain available for shared deployments.
- Workspace assessment pages now surface failed background jobs directly so discovery, graph, simulation, and plan failures are actionable without hunting through job history.
- Release bundles now include the `deploy/` assets needed for local and pilot installation references.

## Verification Scope

The v3.3.0 release workflow published signed images, SBOMs, checksums, and native bundles. Local release validation is documented in `RELEASE.md`; the release workflow validated build, tests, lint, vet, gosec, Trivy filesystem and image scans, Playwright smoke coverage, observability smoke coverage, package matrix, SBOM generation, cosign signing and verification, checksums, and release asset completeness.

## Compatibility

- `/api/v1` remains backward compatible.
- No new `/api/v2` routes were added.
- Shared or production deployments should keep using explicit tenant, service-account, or admin credentials with PostgreSQL-backed state.
- The keyless local session path is intentionally scoped to loopback/local-runtime evaluation.

## Known Limitations

- Connector runtime behavior remains bounded by the support matrix in `docs/reference/support-matrix.md`.
- Fixture-backed tests do not prove production-pilot coverage for every connector pair.
- `:edge` and `:latest` tags remain evaluation conveniences; deployment docs recommend immutable semver tags.
