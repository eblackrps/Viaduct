# Viaduct v1.8.0 Release Notes

Published 2026-04-11.

Viaduct v1.8.0 delivers a substantial frontend and public-surface refresh without changing the backend architecture. This release focuses on a more professional operator experience in the dashboard, a clearer public website, synchronized release and demo assets, and a tighter documentation set around the current discovery-to-report workflow.

## Highlights

- Refreshed the dashboard shell, navigation, page hierarchy, and operator copy so the web UI reads more like a control plane than a demo surface.
- Reworked the workspace-first flow with clearer step guidance, state markers, better empty/loading/error handling, and stronger operator-facing action placement.
- Rewrote the public `site/` landing page, metadata, and social-card text around discovery, dependency mapping, migration planning, supervised execution, and operator-visible reporting.
- Synchronized the repo docs, release notes, screenshots, and package metadata around the same operator workflow and `v1.8.0` release surface.

## Dashboard And Web UI

- The top bar, sidebar, page headers, and section layout now provide a denser and more consistent operator shell.
- Runtime authentication now presents clearer recovery guidance, explicit session-versus-remembered storage messaging, and easier saved-key removal.
- The pilot workspace view now emphasizes progression from authentication to workspace creation, discovery, inspection, simulation, plan save, and report export.
- Inventory, workspace status, readiness, history, and report surfaces now use more consistent cards, badges, operator microcopy, and recovery affordances.

## Public Website

- The GitHub Pages site remains static and custom-domain compatible.
- Landing-page copy now presents Viaduct as an open-source control plane for virtualization migration assessment, planning, and supervised execution.
- Navigation, CTA labels, information architecture, metadata, and trust language now align with the actual product surface instead of older internal strategy phrasing.
- The 404 page and social preview asset now match the revised public-site language.

## Docs And Assets

- Updated the root README, quickstart, dashboard README, docs index, pilot workspace guide, release notes index, and demo presenter materials for the refreshed UI and public positioning.
- Renamed the initial-focus strategy document to `docs/initial-use-case-analysis.md` and updated tracked references accordingly.
- Refreshed the release-facing SVG screenshots so the visuals match the current workspace-first shell and current release labeling.
- Kept the local `examples/lab` path as the default reproducible first-run and demo route.

## Compatibility Notes

- No backend stack rewrite was introduced in this release.
- The Go backend, REST API, CLI, React dashboard in `web/`, and static public site in `site/` all remain in place.
- PostgreSQL remains the recommended non-demo backend for durable operator state.
- The local `examples/lab` workflow remains the default path for first-run validation and demo rehearsal.

## Upgrade Steps

1. Pull the `v1.8.0` bundle or checkout the `v1.8.0` tag.
2. Rebuild or redeploy the CLI, API, dashboard, and static site assets together so the UI copy and release metadata stay aligned.
3. If you serve the dashboard from a non-default origin, confirm `VIADUCT_ALLOWED_ORIGINS` is still set correctly before restarting the API.
4. Rerun the local `examples/lab` flow or your existing supervised pilot smoke path to confirm authentication, workspace creation, job progression, and report export still behave as expected.

## Verification

- `make release-gate` remains the canonical release verification path.
- `make certification-test`, `make plugin-check`, `make contract-check`, and `make package-release-matrix` all remain part of the release-owner validation flow for this release.
- `cd web && npm ci && npm run build` remains the direct dashboard build check.

## Known Issues

- Discovery remains CLI-first. The dashboard is the operator review and control surface over persisted backend state, not a full in-app connection-management flow.
- Live migration behavior outside the documented pilot path should still be validated in a supervised lab or pilot environment before higher-risk use.

## Rollback Notes

- `v1.7.0` remains the immediate rollback target if the UI or docs refresh introduces an operator-facing regression.
- Roll back by redeploying the previous CLI, API, dashboard, and site assets together, then rerunning the lab or pilot smoke path.
- This release does not introduce a documented store migration, so rollback remains operational rather than schema-driven.
