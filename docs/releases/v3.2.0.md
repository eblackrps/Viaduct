# Viaduct v3.2.0 Release Notes

## Docker Install And Verify

Viaduct `v3.2.0` keeps the signed OCI image as the main release artifact.

```bash
docker pull ghcr.io/eblackrps/viaduct:3.2.0
cosign verify ghcr.io/eblackrps/viaduct:3.2.0 \
  --certificate-identity \
  'https://github.com/eblackrps/Viaduct/.github/workflows/image.yml@refs/tags/v3.2.0' \
  --certificate-oidc-issuer \
  'https://token.actions.githubusercontent.com'
cosign verify-attestation --type spdx ghcr.io/eblackrps/viaduct:3.2.0 \
  --certificate-identity \
  'https://github.com/eblackrps/Viaduct/.github/workflows/image.yml@refs/tags/v3.2.0' \
  --certificate-oidc-issuer \
  'https://token.actions.githubusercontent.com'
```

Use immutable semver tags in production. `ghcr.io/eblackrps/viaduct:edge` is still published from merges to `main` and remains non-production.

Native bundles remain attached to the GitHub Release under **Native binaries (alternative)** for environments that cannot run containers.

## Upgrade Notes

- This is a non-breaking minor release. Existing local operator, service account key, tenant key, environment-seeded, and sign-out flows continue to work.
- The dashboard keeps the simpler Get started experience from `v3.1.x`, while `v3.2.0` adds stronger runtime diagnostics and the backend observability baseline behind opt-in configuration.
- `/metrics` remains the stable metrics contract. OpenTelemetry tracing is opt-in through environment variables, and Viaduct still runs normally when no collector is present.

## Highlights

- added backend OpenTelemetry tracing across inbound API requests, workspace jobs, discovery, planning and simulation, migration execution, report export, connector HTTP clients, and the PostgreSQL-backed persistence path that operators use during real assessments
- shipped a local Grafana + Tempo stack plus source-controlled provisioning and a validation helper so release owners can prove traces arrive from a live Viaduct runtime before tagging
- upgraded `viaduct doctor` and `viaduct status --runtime` so users can see config validity, active store status, auth status, and concrete readiness degradation reasons instead of a simple reachable/not-reachable split
- strengthened the real runtime golden-path smoke from `viaduct start` through local session, assessment creation, discovery, dependency graph, simulation, saved plan, and exported report
- refreshed the root docs, install guides, public site, deployment samples, release references, and README/demo screenshots so the shipped workflow matches the current product surface

## Validation Highlights

- `go test ./...` passed
- `go run ./scripts/release_surface_check` passed
- the GitHub PR validation path passed with CI, Playwright, CodeQL, image, and observability smoke coverage green before the release cut
- the checked-in README and demo screenshots were regenerated from the current dashboard flow, including graph and reporting captures

## Native Binaries (Alternative)

If you cannot run containers, the release still publishes signed native bundles for:

- `linux/amd64`
- `linux/arm64`
- `darwin/arm64`
- `windows/amd64`

These bundles are generated from the same tagged source and remain covered by release checks, but the signed OCI image remains the main packaged install path.
