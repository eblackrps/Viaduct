# Viaduct v2.7.0 Release Notes

## Upgrade Notes

- `VIADUCT_ADMIN_KEY` now stores the persisted `sha256:<hex>` digest, not the plaintext admin secret presented in `X-Admin-Key`.
- loopback trust is stricter again around malformed or zone-qualified forwarded client IP headers, and local-runtime rejection auditing now covers more dashboard paths.
- PostgreSQL credential migration now runs in two phases. Viaduct seeds durable credential-hash rows first, then clears legacy plaintext only after it verifies the expected hash rows are present.
- the release workflow validates the expected bundle set and verifies published bundle checksums before it reuses the packaged binaries in the container-image build.

## Highlights

- tightened dashboard-session revocation coordination and added stronger concurrent lookup coverage around session replay handling
- continued hardening credential-hash comparison, malformed-hash handling, and credential-rotation audit coverage
- removed the workspace-executor enqueue acknowledgement race, added a typed result-channel failure path, and strengthened timeout handling in queued-work paths
- refreshed the focus trap and dashboard request dedupe paths so drawer teardown and identical opt-in GET requests were more resilient
- tightened the release workflow around explicit artifact manifests, stronger signing-identity validation, and source-controlled bundle verification
- forward-looking platform work discussed during the v2.7.0 cycle now lives in [`docs/releases/roadmap.md`](./roadmap.md) instead of the published release notes

## Breaking Changes

- deployments must provision `VIADUCT_ADMIN_KEY` as a stored `sha256:<hex>` digest instead of the plaintext admin key
- malformed `X-Forwarded-For` or `X-Real-IP` values are now rejected more aggressively during strict loopback enforcement
- operators with partially migrated PostgreSQL credentials should expect startup to stop before clearing plaintext if the prerequisite hash rows are missing
