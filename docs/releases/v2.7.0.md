# Viaduct v2.7.0 Release Notes

## Upgrade Notes

- `VIADUCT_ADMIN_KEY` now stores the persisted `sha256:<hex>` digest, not the plaintext admin secret presented in `X-Admin-Key`.
- loopback trust is stricter again: malformed or zone-qualified forwarded client IP headers are rejected without falling back to `X-Real-IP`, and loopback rejection audits now cover rejected `GET` requests too.
- PostgreSQL credential migration now runs in two phases. Viaduct seeds durable credential-hash rows first, then clears legacy plaintext only after it verifies the expected hash rows are present.
- the release workflow now validates the exact expected bundle set and verifies published bundle checksums before it reuses the packaged binaries in the container-image build.

## Highlights

- made dashboard-session revocation safer by keeping the in-memory session entry intact when the durable revocation write fails and by coordinating revocation reads more tightly with the session-manager lock domain
- normalized malformed and zero credential hashes onto the same constant-time comparison path, and enriched credential-rotation audit events with both the old and new hash prefixes
- removed the workspace-executor enqueue acknowledgement race, added a typed result-channel failure path, and strengthened timeout coverage so failed enqueue attempts do not inflate queue depth
- refreshed the focus trap so it always uses the latest Escape callback and recovers focus safely even if the trigger disappears or the trap root unmounts
- kept opt-in GET dedupe efficient in the dashboard client while ensuring the in-flight dedupe map drains completely once the final waiter settles
- tightened the release workflow and its regression coverage around explicit artifact manifests, tag-pinned signing identities, and source-controlled manifest validation so release owners can prove the published bundle set before tagging

## Breaking Changes

- deployments must provision `VIADUCT_ADMIN_KEY` as a stored `sha256:<hex>` digest instead of the plaintext admin key
- malformed `X-Forwarded-For` or `X-Real-IP` values are now rejected more aggressively during strict loopback enforcement
- operators with partially migrated PostgreSQL credentials should expect startup to stop before clearing plaintext if the prerequisite hash rows are missing
