# Viaduct v1.6.0 Release Notes

Published 2026-04-11.

Viaduct v1.6.0 turns the current repository into a more cohesive operator application for the assessment and supervised pilot focus without rewriting the stack.

## Highlights

- Added a first-class pilot workspace model that persists source connections, discovery snapshots, dependency graph output, target assumptions, readiness results, saved plans, approvals, notes, and exported reports in one operator-owned record.
- Added persisted background jobs for workspace discovery, graph generation, simulation, and plan generation so the dashboard and API can replay the same stateful workflow.
- Reworked the dashboard around a runtime authentication bootstrap flow and a workspace-first experience: create workspace, discover, inspect, simulate, save plan, export report.
- Added a deterministic `examples/lab` smoke path that exercises the first successful workspace-to-report flow end to end.

## Operator Workflow

The default operator path now starts on the pilot workspace route and follows this sequence:

1. Bootstrap dashboard access with a service-account key or tenant key.
2. Create a pilot workspace.
3. Discover and save source snapshots into the workspace.
4. Inspect the workload set and dependency graph.
5. Run readiness simulation.
6. Save a dry-run migration plan.
7. Export a pilot report.

This keeps the current story narrow and coherent: assessment, readiness reduction, approval-ready planning, and supervised pilot handoff.

## API And Persistence

- New tenant-scoped workspace routes provide create, list, get, patch, job status, and report export surfaces.
- Workspace jobs persist input, output, status, retryability, operator-facing messages, and request correlation IDs.
- Saved plans persist dry-run migration state through the existing migration store path rather than a parallel planning stack.
- PostgreSQL remains the recommended non-demo backend for durable workspace state.

## Dashboard And Auth

- The dashboard now accepts runtime credentials through a bootstrap screen instead of relying only on build-time environment variables.
- Service-account keys remain the preferred normal operator credential.
- Tenant keys remain the bootstrap and break-glass credential.
- Loading, empty, retry, and error states now carry richer operator-facing detail for the workspace flow.

## Release Artifacts

- GitHub release bundles are available for Linux and Windows operator workstations.
- The published release remains aligned to `make release-gate` as the canonical verification path.
- The local `examples/lab` flow remains the default reproducible demo and evaluation path.

## Lab And Verification

- The local `examples/lab` path remains the default demo and evaluation route.
- New example payloads under `examples/lab/` cover tenant bootstrap, service-account creation, and seeded pilot workspace intake.
- The focused workspace smoke is backed by `tests/integration/pilot_workspace_smoke_test.go`.
- `make release-gate` remains the canonical release verification path.

## Compatibility Notes

- No stack rewrite: the Go backend, REST API, CLI, React dashboard in `web/`, and separate public site in `site/` remain in place.
- The product story stays bounded to the current assessment and supervised pilot focus.
- Live migration execution should still be validated in a supervised lab or pilot before it is treated as routine automation.
