# Viaduct v3.0.0 Release Notes

## Docker Install And Verify

Viaduct `v3.0.0` makes the signed OCI image the canonical release artifact.

```bash
docker pull ghcr.io/eblackrps/viaduct:3.0.0
cosign verify ghcr.io/eblackrps/viaduct:3.0.0 \
  --certificate-identity \
  'https://github.com/eblackrps/viaduct/.github/workflows/image.yml@refs/tags/v3.0.0' \
  --certificate-oidc-issuer \
  'https://token.actions.githubusercontent.com'
cosign verify-attestation --type spdx ghcr.io/eblackrps/viaduct:3.0.0 \
  --certificate-identity \
  'https://github.com/eblackrps/viaduct/.github/workflows/image.yml@refs/tags/v3.0.0' \
  --certificate-oidc-issuer \
  'https://token.actions.githubusercontent.com'
```

Use immutable semver tags in production. `ghcr.io/eblackrps/viaduct:edge` is published from merges to `main` and is explicitly not for production use.

Native bundles remain attached to the GitHub Release under **Native binaries (alternative)** for environments that cannot run containers.

## Upgrade Notes

- Docker is now the canonical install and upgrade path. Container deployments should mount writable state at `/var/lib/viaduct`, run with a read-only root filesystem where practical, and use `/healthz` for liveness plus `/readyz` for readiness.
- `VIADUCT_ADMIN_KEY` now accepts either plaintext or `sha256:<hex>` storage formats. The hashed form is the recommended long-term configuration and is the one documented for production deployments.
- The release model changed from continuous per-PR cuts to milestone-based minor releases. Patch releases are now reserved for shipped regressions.
- The dashboard intentionally ships as a complete light-mode surface in `v3.0.0`. Partial dark mode is not part of this release.

## Highlights

- completed the remaining v2.7.0 hardening follow-up work around admin-key handling, malformed credential-hash decoding, forwarded-header rejection logging, durable audit retry handling, and queue-accounting coverage
- promoted the GHCR image to the canonical artifact with a dedicated GitHub Actions pipeline for build, signature verification, SBOM attestation, provenance, and image scanning
- refreshed the operator console visual system around shared typography, radius, and color tokens, plus route-level accessibility checks and Playwright visual-regression coverage for the primary operator routes
- aligned public docs, deployment samples, and the roadmap around the v3 release model so the changelog records shipped work and forward-looking platform features live in roadmap docs instead

## Breaking Changes

- operators should treat the OCI image as the primary production artifact going forward; native binary bundles remain supported but are now an alternative path rather than the canonical install surface
- release automation now publishes the mutable `:edge` tag from `main`, but production guidance expects pinned semver tags and signature verification before rollout

## Native Binaries (Alternative)

If you cannot run containers, the release still publishes signed native bundles for:

- `linux/amd64`
- `linux/arm64`
- `darwin/arm64`
- `windows/amd64`

These bundles are generated from the same tagged source and remain covered by the release verification path, but they are no longer the primary operator story in the docs or release body.
